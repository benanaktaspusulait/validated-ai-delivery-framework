# T3.2 — Experiment Mode Assignment

**Phase:** 3  
**Owner:** Platform Engineer  
**Estimate:** 1 day  
**Depends on:** T3.1

## Objective

Implement random assignment of PRs to treatment/control groups for A/B testing.

## Acceptance criteria

- [ ] Each new PR randomly assigned to treatment (50%) or control (50%).
- [ ] Assignment is deterministic (PR number mod 100).
- [ ] Assignment stored in pull_requests table (experiment_group field).
- [ ] Treatment group receives PR comments.
- [ ] Control group receives no PR comments.
- [ ] Assignment is persisted and reproducible.

## Technical notes

```text
Schema change (migration):
  ALTER TABLE pull_requests ADD COLUMN experiment_group TEXT;
  -- Values: 'treatment', 'control', NULL (not in experiment)

Assignment logic:
  function assignExperimentGroup(pr_number: int) -> string:
    hash = pr_number % 100
    if hash < 50:
      return 'treatment'
    else:
      return 'control'

  Called when PR is first processed (in ETL or webhook handler).
  Stored: pull_requests.experiment_group = assignExperimentGroup(pr.external_id)

PR Comment Bot integration:
  In T3.1, add a check before posting:
    if pull_requests.experiment_group = 'control':
      skip comment (control group receives no guidance)
    if pull_requests.experiment_group = 'treatment':
      post comment as normal
    if pull_requests.experiment_group IS NULL:
      post comment (not in experiment = treated as treatment)

Filtering for A/B analysis:
  SELECT experiment_group, COUNT(*)
  FROM pull_requests
  WHERE repository_id IN (team's repos)
    AND created_at_source >= experiment_start_date
  GROUP BY experiment_group
  -- Expect ~50/50 split
```

## Test cases

```text
Unit tests:
  - PR #100 -> 100 % 100 = 0 -> treatment
  - PR #150 -> 150 % 100 = 50 -> control
  - PR #201 -> 201 % 100 = 1 -> treatment
  - 1000 PRs -> ~500 treatment, ~500 control (within 5% variance).

Integration test:
  - Process 100 PRs -> ~50 in each group.
  - Re-process same PRs -> same assignment (deterministic).
```
