# T0.4 — Branch Protection Registration

**Phase:** 0  
**Owner:** Platform Engineer  
**Estimate:** 1 hour  
**Depends on:** T0.1

## Objective

Register (but NOT activate) a branch protection rule for the main branch in pilot repositories. This ensures the rule exists for Phase 4 activation when the AI Metadata Blocker goes live.

## Step-by-step

```text
Step 1: Navigate to branch protection
  For each pilot repo:
    1. Go to repo > Settings > Branches
    2. Click "Add branch protection rule"

Step 2: Configure the rule
  Branch name pattern: main

  Configure these options (but leave them UNCHECKED for now):
    [ ] Require a pull request before merging
        [ ] Required approvals: 1
        [ ] Dismiss stale pull request approvals when new commits are pushed
    [ ] Require status checks to pass before merging
        [ ] Status checks to require: (will add "AI Metadata Enforcement" in Phase 4)
    [ ] Require conversation resolution before merging
    [ ] Require signed commits
    [ ] Require linear history
    [ ] Include administrators (leave unchecked — admins can still bypass)

  Leave ALL checkboxes UNCHECKED. The rule exists but has no active requirements.

Step 3: Save the rule
  Click "Create" or "Save changes".
  The rule is now registered but inactive.

Step 4: Document
  Record in [Phase0]_config_changes.yaml:

    branch_protection:
      repos:
        - owner: example-org
          repo: payments-api
          branch: main
          status: registered_not_activated
          options_configured:
            - require_pull_request: false
            - require_status_checks: false
            - require_conversation_resolution: false
          enforcement_phase: 4
          notes: "Rule exists but has no active requirements. Will activate in Phase 4 with AI Metadata Enforcement check."

Step 5: Verify
  For each pilot repo:
    1. Go to Settings > Branches
    2. Confirm the "main" rule exists.
    3. Create a test PR and merge it WITHOUT any checks.
    4. Confirm it merges successfully (rule is inactive).
    5. Delete the test PR.
```

## Acceptance criteria

- [ ] Branch protection rule added for `main` in each pilot repo.
- [ ] Rule is saved but inactive (no requirements enforced).
- [ ] A PR can still be merged without any checks passing.
- [ ] Configuration documented in [Phase0]_config_changes.yaml.
- [ ] Rule is ready for Phase 4 activation (just needs checkboxes enabled + check added).

## Validation

```text
1. Branch protection rule exists in repo Settings > Branches.
2. A test PR can be merged without any checks.
3. The rule is documented in config_changes.yaml.
4. No production workflow is affected (rule is inactive).
```

## Common issues

```text
- "Cannot create rule": you need admin access to the repo.
- "Rule already exists": there may be an existing rule. Edit it instead of creating new.
- "Main branch doesn't exist": the default branch might be "master". Use that name instead.
```
