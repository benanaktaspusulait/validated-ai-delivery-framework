# T0.6 — Legal/Privacy Review

**Phase:** 0  
**Owner:** Legal or privacy counsel  
**Estimate:** 1 week  
**Depends on:** None

## Objective

Obtain written legal/privacy approval for collecting and processing developer activity data.

## What to present to legal

```text
Prepare a 2-page brief for legal/privacy counsel:

1. DATA WE COLLECT:
   - PR metadata: title, author (pseudonymised), files changed, lines changed, timestamps
   - Review data: reviewer (pseudonymised), timestamps, comments count, approval status
   - Jira data: issue key, type, severity, status, linked PR
   - AI metadata: usage types checkbox (not free text), confidence level
   - Derived metrics: team-level aggregates only

2. DATA WE DO NOT COLLECT:
   - Raw prompt content (never stored)
   - Individual performance scores (prohibited by design)
   - IDE telemetry (not in MVP)
   - Code content (only metadata about code changes)

3. RETENTION:
   - Detailed data: 12 months, then purged
   - Aggregated metrics: 24 months
   - Audit logs: 24 months

4. PSEUDONYMISATION:
   - Developer identifiers (GitHub login, email) are hashed before storage
   - Platform Lead and Data Steward can reverse the hash for debugging
   - Managers see only team-level data, never individual identifiers

5. USAGE:
   - Team-level delivery health monitoring
   - NOT individual performance evaluation
   - NOT performance reviews or promotions
   - Policy: "no individual performance use" enforced in the platform

6. LEGAL BASIS:
   - Legitimate interest (improving delivery processes)
   - Or consent (depending on jurisdiction)
   - GDPR: data minimisation, purpose limitation, storage limitation
```

## What legal should confirm

```text
- [ ] Data scope is acceptable under applicable privacy law (GDPR, CCPA, etc.)
- [ ] Retention periods (12/24 months) are acceptable
- [ ] Pseudonymisation approach is sufficient
- [ ] "No raw prompt storage" is compliant
- [ ] Legitimate interest or consent basis is valid
- [ ] Data Subject Access Request (DSAR) process is defined
- [ ] Right to erasure is supported (purge job in Phase 5)
- [ ] Any additional requirements or restrictions

If legal requires changes:
  - Update docs/governance-and-privacy.md
  - Update data retention in data-model.md
  - Document changes in [Phase0]_config_changes.yaml
```

## Acceptance criteria

- [ ] Legal/privacy counsel has reviewed the data scope.
- [ ] Written approval received and stored.
- [ ] Data retention policy (12 months detailed, 24 months aggregates) is acceptable.
- [ ] Pseudonymisation approach for developer identifiers is approved.
- [ ] No raw prompt storage is confirmed as compliant.
- [ ] Any legal requirements documented and incorporated.
- [ ] Approval documented in [Phase0]_exit_report.pdf.
