# T0.5 — Psychological Safety Baseline

**Phase:** 0  
**Owner:** Pilot-team Engineering Manager  
**Estimate:** 2 hours (survey + analysis)  
**Depends on:** None

## Objective

Measure the pilot team's psychological safety baseline using the 6-question pulse survey. The score must be >= 3.5 to proceed to Phase 1.

## Step-by-step

```text
Step 1: Prepare the survey
  Create an anonymous survey (Google Forms, Typeform, or paper):

  Title: "AI Delivery Pilot — Team Readiness Survey"
  Introduction: "We are evaluating whether this team is ready to pilot an AI delivery dashboard. 
  The dashboard tracks team-level delivery health, never individual performance. 
  Your responses help us understand whether the team feels safe participating. 
  All responses are anonymous."

  Questions (1-5 scale: 1 = strongly disagree, 5 = strongly agree):

  Q1: I feel safe declaring AI usage in my work.
  Q2: I feel safe challenging AI-generated output.
  Q3: I feel safe saying I do not understand AI-generated code.
  Q4: Our team reviews AI-assisted code carefully.
  Q5: AI metrics are used to improve the system, not blame individuals.
  Q6: I do not feel pressured to use AI when it is not helpful.

  Settings:
  - Anonymous responses (no email collection)
  - One response per person
  - Required: all 6 questions

Step 2: Brief the team (30 minutes before survey)
  Hold a 30-minute meeting:
    1. Purpose (5 min): "We are not measuring individual productivity. We are validating 
       whether the system can use AI safely."
    2. What the platform does (5 min): team-level metrics, no personal dashboards, no rankings.
    3. What it does NOT do (5 min): no individual scoring, no raw prompt storage, no IDE monitoring.
    4. Your role (5 min): fill in PR template, read in-PR guidance, give feedback.
    5. The rules (5 min): non-negotiable principles (no ranking, no raw prompts, no hard enforcement before Phase 4).
    6. Q&A (5 min).

  Key message: "This survey is anonymous. There are no right or wrong answers. 
  Your honest feedback helps us decide whether to proceed."

Step 3: Distribute the survey
  Send the survey link immediately after the briefing.
  Deadline: 24 hours.
  Follow up with anyone who hasn't responded.

Step 4: Collect and calculate
  After deadline:
    1. Export responses.
    2. Calculate average for each question.
    3. Calculate overall average = (Q1_avg + Q2_avg + ... + Q6_avg) / 6.

  Example:
    Q1: 4.2, Q2: 3.8, Q3: 3.5, Q4: 4.0, Q5: 4.1, Q6: 3.9
    Overall: (4.2 + 3.8 + 3.5 + 4.0 + 4.1 + 3.9) / 6 = 3.92

Step 5: Analyze results
  If overall >= 3.5: PROCEED. Record the score.
  If overall < 3.5: PAUSE. See fail action.

  Also analyze individual questions:
    - If Q2 ("challenge AI output") is low: developers feel pressured to accept AI code.
    - If Q3 ("understand AI code") is low: developers are submitting code they don't understand.
    - If Q5 ("not for blame") is low: trust issue with management.
    - If Q6 ("not pressured") is low: adoption pressure is too high.

Step 6: Record results
  Document in [Phase0]_exit_report.pdf:
    - Survey method (anonymous, online/paper)
    - Number of respondents
    - Individual question scores
    - Overall average
    - Pass/fail determination
    - Any concerns noted (from open-ended question if included)
```

## Acceptance criteria

- [ ] Anonymous survey created with all 6 questions.
- [ ] 30-minute briefing held with all pilot team members.
- [ ] Survey distributed within 24 hours of briefing.
- [ ] 100% participation (all team members responded).
- [ ] Average score calculated: overall + per-question.
- [ ] Score is >= 3.5. If not, see fail action.
- [ ] Results documented in [Phase0]_exit_report.pdf.

## Validation

```text
1. Survey link sent to all team members.
2. All responses collected (anonymously).
3. Average score >= 3.5.
4. Individual question scores documented.
5. Methodology and results in exit report.
```

## Fail action

```text
If score < 3.5:
1. Run a blameless retro to understand concerns.
   - "What would need to change for you to feel safe participating?"
   - "What specifically concerns you about the dashboard?"
2. Address the top concern:
   - If Q5 is low: re-communicate "no individual performance use" more strongly.
   - If Q6 is low: clarify that AI usage is optional, not mandatory.
   - If Q2/Q3 are low: explain that the platform helps, not punishes.
3. Re-administer after 1 week.
4. If still < 3.5: do not proceed to Phase 1. Cancel pilot or reduce scope.

Document the retro and re-test results in the exit report.
```
