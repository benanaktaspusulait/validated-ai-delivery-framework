# T0.8 — Pilot Team Briefing

**Phase:** 0  
**Owner:** Platform Lead  
**Estimate:** 1 hour  
**Depends on:** T0.6, T0.7

## Objective

Brief the pilot team on the platform's purpose, data usage and non-negotiable cultural rules. This is the team's first exposure to the project.

## Step-by-step

```text
Step 1: Prepare (30 min before)
  - Book a 30-minute meeting with the full pilot team.
  - Prepare the slide deck or talking points (see agenda below).
  - Have the PR template ready to show.
  - Have the psychological safety survey link ready.

Step 2: Run the briefing (30 minutes)

  Agenda:
    1. Purpose (5 min)
       "We are evaluating whether this team is ready to pilot an AI delivery dashboard.
        We are NOT measuring individual productivity. We ARE validating whether the 
        system can use AI safely, sustainably and effectively."

    2. What the platform does (5 min)
       - Collects PR metadata (title, files, timestamps, AI usage checkboxes)
       - Computes team-level metrics (AI PR rate, review debt, defect rate)
       - Shows a dashboard for team leads and managers
       - No personal dashboards. No rankings. No scores for individuals.

    3. What the platform does NOT do (5 min)
       - No individual scoring or rankings
       - No raw prompt storage
       - No IDE monitoring
       - No performance reviews
       - No "who uses AI most" leaderboard

    4. Your role as developers (5 min)
       - Fill in the PR template honestly (AI usage checkboxes)
       - Read the in-PR guidance (comments from the bot)
       - Give feedback in retrospectives
       - You can challenge any metric interpretation

    5. The rules (5 min)
       - Non-negotiable principles (from docs/governance-and-privacy.md):
         * No individual AI productivity ranking
         * No use in performance reviews
         * Team-level reporting by default
         * Blameless retrospectives
         * Developers can challenge metric interpretation
         * AI usage disclosure must not be punished

    6. Show the PR template (2 min)
       - Open a pilot repo > New pull request
       - Show the template: AI checkboxes, confidence, ownership confirmation
       - "This is what you'll see when you create a PR."

    7. Q&A (3 min)
       - Open floor for questions.
       - Common questions:
         * "Will this affect my performance review?" -> No. Absolutely not.
         * "Do I have to use AI?" -> No. AI usage is optional.
         * "What if I don't fill in the template?" -> In Phase 3, you'll get a 
           gentle reminder. In Phase 4, it will be required before merge.
         * "Who sees my data?" -> Only the platform team and team leads. 
           Managers see team-level aggregates only.

Step 3: Distribute the survey
  Immediately after the briefing:
  "We'd like you to fill in a short anonymous survey about how safe you feel 
   participating. The link is in Slack/email. Please respond within 24 hours.
   There are no right or wrong answers."

  Survey link: (from T0.5)

Step 4: Record attendance
  - Note who attended (for the exit report).
  - Note any questions or concerns raised.
  - Note any follow-up actions needed.
```

## Acceptance criteria

- [ ] 30-minute briefing session held with all pilot team members.
- [ ] Purpose explained: "We are not measuring individual productivity."
- [ ] Data usage explained: team-level only, no individual scoring.
- [ ] Non-negotiable rules communicated (no ranking, no raw prompts, no hard enforcement before Phase 4).
- [ ] PR template walkthrough completed.
- [ ] Psychological safety survey distributed.
- [ ] Q&A session held.
- [ ] Attendance recorded for [Phase0]_exit_report.pdf.
- [ ] Any concerns or follow-up actions documented.

## Validation

```text
1. Meeting held with all pilot team members.
2. All key messages delivered (purpose, what it does/doesn't do, rules, role).
3. PR template shown and explained.
4. Survey link distributed.
5. Questions answered.
6. Attendance and notes in exit report.
```
