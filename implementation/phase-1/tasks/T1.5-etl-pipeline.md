# T1.5 — ETL Pipeline

**Phase:** 1  
**Owner:** Platform Engineer  
**Estimate:** 3 days  
**Depends on:** T1.1, T1.2, T1.3, T1.4

## Objective

Build ETL pipeline that transforms raw_events into normalized tables (pull_requests, review_analytics, jira_issues).

## Acceptance criteria

- [ ] ETL processes raw_events with event_type starting with "pull_request" into `pull_requests` table.
- [ ] ETL processes raw_events with event_type starting with "pull_request_review" into `review_analytics` table.
- [ ] ETL processes raw_events with event_type starting with "jira" into `jira_issues` table.
- [ ] AI metadata extracted from PR body checkboxes.
- [ ] PRs linked to repositories via repo mapping.
- [ ] Jira issues linked to PRs via commit messages or branch names.
- [ ] Processing status updated: pending -> processing -> completed (or retry/dead_letter).
- [ ] ETL is idempotent (re-processing same raw_event does not create duplicates).

## Technical notes

```text
ETL can be:
  - Event-driven: triggered by new raw_events (preferred for real-time).
  - Batch: scheduled job every 15 minutes (simpler for pilot).

Processing flow:
1. SELECT * FROM raw_events WHERE processing_status = 'pending' LIMIT 1000.
2. For each event:
   a. Set processing_status = 'processing'.
   b. Parse payload based on event_type.
   c. Extract fields and INSERT/UPDATE the target table.
   d. Set processing_status = 'completed'.
   e. On error: set processing_status = 'retry', increment retry count.
      After 3 retries: set processing_status = 'dead_letter'.

PR normalization (pull_requests table):
  From pull_request events:
    external_id = PR number
    repository_id = lookup from repositories table (by owner/name)
    title = payload.title
    author_id = pseudonymise(payload.user.login)
    created_at_source = payload.created_at
    merged_at_source = payload.merged_at
    closed_at_source = payload.closed_at
    changed_files = payload.changed_files
    changed_lines = payload.additions + payload.deletions

  AI metadata extraction from PR body:
    if body contains "[x] No AI assistance used" -> ai_assisted = false
    if body contains "[x] AI used for ..." -> ai_assisted = true
      Extract ai_usage_types = list of matched categories
    if body contains "[x] High" -> ai_assistance_confidence = "high"
    if body contains "[x] Medium" -> ai_assistance_confidence = "medium"
    if body contains "[x] Low" -> ai_assistance_confidence = "low"
    If neither declared -> ai_assisted = NULL (unknown)

Review normalization (review_analytics table):
  From pull_request_review events:
    pull_request_id = lookup from pull_requests (by external_id)
    first_review_at = MIN(review_submitted_at) for this PR
    approved_at = review submitted with state = 'APPROVED'
    comments_count = COUNT review comments
    reviewer_count = COUNT distinct reviewers

Jira normalization (jira_issues table):
  From jira:issue_created/updated events:
    external_key = issue key (e.g. PILOT-123)
    issue_type = payload.fields.issuetype.name
    severity = payload.fields.priority.name
    status = payload.fields.status.name
    created_at_source = payload.fields.created
    resolved_at_source = payload.fields.resolutiondate

  Linking PRs to Jira issues:
    Option 1: Parse commit messages for Jira key pattern (e.g. PILOT-123)
    Option 2: Parse branch name for Jira key pattern
    Option 3: Use Jira issue link type "is developed by"
    Set linked_pull_request_id = matching PR's id
```

## Validation

```text
1. Process 100 raw GitHub events -> pull_requests and review_analytics populated.
2. Process 50 raw Jira events -> jira_issues populated.
3. AI metadata correctly extracted from PR bodies.
4. Re-processing same events -> no duplicates.
5. Malformed events -> processing_status = 'dead_letter', processing_error logged.
```
