# T4.2 — Emergency Override Flow

**Phase:** 4  
**Owner:** Platform Engineer  
**Estimate:** 1 day  
**Depends on:** T4.1

## Objective

Implement the emergency override flow including logging, rate limiting and retro triggers.

## Acceptance criteria

- [ ] "emergency-fix" label bypasses all blocking checks.
- [ ] Override logged to policy_overrides table.
- [ ] Comment posted acknowledging override.
- [ ] Soft limit: after 3 overrides/sprint, second approver required.
- [ ] Hard limit: after 5 overrides/sprint, further overrides blocked.
- [ ] Retro triggered at 3 overrides/sprint.

## Technical notes

```text
Override detection (in AI Metadata Enforcement workflow):
  if labels.includes("emergency-fix"):
    1. Post comment: "Emergency override recorded. A retrospective will be scheduled if this exceeds 3 per sprint."
    2. Log: POST /api/v1/policy-overrides { pull_request_id, policy_name, override_label, overridden_by, reason }
    3. Skip all blocking checks. Return.

Rate limiting (in platform API, POST /api/v1/policy-overrides):
  Before inserting the override:
    SELECT COUNT(*) FROM policy_overrides
    WHERE pull_request_id IN (SELECT id FROM pull_requests WHERE repository_id IN (team's repos))
      AND override_label = 'emergency-fix'
      AND created_at >= (start_of_current_sprint)

    If count >= 5 (hard limit):
      Return 429 with error: "Emergency override limit reached for this sprint. Contact Platform Lead."
      Do NOT record the override.

    If count >= 3 (soft limit):
      Add to override record: requires_second_approver = true.
      Post comment: "This is the {N}th emergency override this sprint. A second approver (tech lead or platform admin) is required."
      The override still goes through, but is flagged for retro.

Retro trigger:
  When count hits 3:
    Notify Platform Lead: "Team {name} has used 3 emergency overrides this sprint. Schedule a 15-minute retro."
    Provide retro questions:
      1. Are we using emergency-fix correctly?
      2. Is there a workflow improvement that would prevent future emergencies?
      3. Should our policy thresholds be adjusted?
```

## Test cases

```text
- Override with "emergency-fix" label -> logged, check skipped.
- 3rd override in sprint -> retro notification sent.
- 5th override in sprint -> 429 returned, override blocked.
- Override without label -> normal check flow.
```
